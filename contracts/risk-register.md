### RISK-001  -  Stripe billing_invoices non-idempotent insert

| Field | Value |
|-------|-------|
| **ID** | RISK-001 |
| **Severity** | **P0** |
| **Status** | Open |
| **Production-blocking** | Yes |
| **Date Logged** | 2026-03-16 |
| **Logged By** | Team 3 Module 1 audit |
| **Owner** | Backend / Billing team |
| **Source File** | `app/api/webhooks/stripe/route.ts` |

**Description:**
`billing_invoices` inserts triggered by `invoice.paid` and `invoice.payment_failed` Stripe webhook events use a plain `.insert()` with no `onConflict` clause. Stripe guarantees at-least-once delivery, meaning the same event can be delivered more than once (e.g., on network timeout before a 200 is returned). Each duplicate delivery creates a new row in `billing_invoices`, resulting in phantom invoice records that misrepresent billing history.

The `subscriptions` and `tenants` table updates in the same handler are natural upserts and are not affected by this gap.

**Impact:**
- Duplicate rows in `billing_invoices` for the same Stripe invoice
- Incorrect invoice counts and amounts on any billing history UI
- Revenue reporting errors

**Remediation:**
1. Add a unique constraint on `billing_invoices.stripe_invoice_id` (migration required)
2. Change the `.insert()` to `.upsert()` with `onConflict: 'stripe_invoice_id'`

```sql
-- Migration (add to next numbered migration file)
ALTER TABLE billing_invoices
  ADD CONSTRAINT billing_invoices_stripe_invoice_id_key UNIQUE (stripe_invoice_id);
```

```typescript
// Route change
await supabase
  .from('billing_invoices')
  .upsert({ stripe_invoice_id: invoice.id, ... }, { onConflict: 'stripe_invoice_id' })
```

**Approval required:** Yes  -  requires a schema migration. Must go through the standard migration approval process.

### RISK-002  -  GHL and Clio OAuth adapters have no token refresh

| Field | Value |
|-------|-------|
| **ID** | RISK-002 |
| **Severity** | **P0** |
| **Status** | **DEFERRED  -  2026-03-16** (GHL/Clio not in launch scope) |
| **Production-blocking** | Not blocking immediate launch. Blocks GHL/Clio go-live when those surfaces are enabled. |
| **Date Logged** | 2026-03-16 |
| **Date Deferred** | 2026-03-16 |
| **Logged By** | Team 3 Module 1 audit |
| **Owner** | Integration team |
| **Source Files** | `lib/services/ghl/client.ts`, `lib/services/ghl/oauth.ts`, `lib/services/clio/client.ts`, `lib/services/clio/oauth.ts` |

**Description:**
The GHL and Clio OAuth adapters store access tokens and refresh tokens at connection time. Neither adapter implements a token refresh interceptor. OAuth access tokens for both platforms expire (GHL: 24 hours; Clio: 1 hour for access tokens). After expiry, all API calls made with the stale access token will return 401. There is no code path that detects a 401, uses the stored refresh token to obtain a new access token, and retries the original request.

The result is that integrations will appear to work after initial connection and silently begin failing after the first token expiry window. Users will not be informed unless they actively check the integration status.

**Impact:**
- GHL contact and opportunity syncs silently fail after 24 hours
- Clio matter and contact imports silently fail after 1 hour
- No user-facing alert generated by the current integration code
- Operations team will likely discover this through support complaints, not monitoring

**Remediation:**
Add a token refresh interceptor to both adapters. The interceptor should:
1. Catch 401 responses from the platform API
2. Use the stored `refresh_token` from `platform_connections` to call the platform's token endpoint
3. Store the new `access_token` and updated `expires_at` back to `platform_connections`
4. Retry the original request once with the new token
5. If the refresh itself fails (e.g., refresh token expired or revoked), mark the `platform_connections` row as `status = 'disconnected'` and surface a reconnection prompt to the tenant admin

**Approval required:** Yes  -  modifies existing integration adapter files (core file edit). Requires separate controlled approval. Does not require a schema migration (the `platform_connections` table already has `refresh_token` and `expires_at` columns, per Module 1 audit).

### RISK-003  -  Microsoft Graph `graphFetch` has unbounded recursive retry

| Field | Value |
|-------|-------|
| **ID** | RISK-003 |
| **Severity** | **P0** |
| **Status** | **CLOSED  -  2026-03-16** |
| **Production-blocking** | Resolved |
| **Date Logged** | 2026-03-16 |
| **Date Closed** | 2026-03-16 |
| **Logged By** | Team 3 Module 1 audit |
| **Closed By** | Team 3 P1c operator lane |
| **Owner** | Integration team |
| **Source File** | `lib/services/microsoft-graph.ts` |

**Description:**
The `graphFetch` helper used to make Microsoft Graph API calls implements retry logic by calling itself recursively. There is no maximum recursion depth, no attempt counter, and no circuit breaker. If Microsoft Graph returns repeated 429 (rate-limited) or 5xx responses, the function recurses indefinitely until the call stack is exhausted.

**Resolution:**
`lib/services/microsoft-graph.ts` rewritten with an iterative `while(true)` loop bounded by `MAX_GRAPH_RATE_LIMIT_RETRIES = 5` (line 239). `Retry-After` header read and honoured via `setTimeout`. Non-429 4xx responses throw `GraphError` immediately with no retry. Zero recursive calls  -  stack overflow is impossible by construction.

**Runtime proof  -  2026-03-16T16:51:04Z:**
1. Repeated 429s → `GraphError(429, "Graph API rate limit exhausted after 5 retries")` thrown on attempt 6. **PASS**
2. `Retry-After: 2` honoured → elapsed 2001ms. **PASS**
3. Non-429 4xx (400/401/403/404/405/422) all threw immediately at 0ms. **PASS**
4. Normal 200 path returned correctly at 0ms. **PASS**
